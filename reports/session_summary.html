Context

What this project is

The team is building an MVP web platform for hotels: analyze historical review data, surface actionable insight for hotel managers, and train a model — starting with regression — that predicts and explains a hotel's average score, so managers know which specific things to improve. Data is hosted on Neon (serverless Postgres, region us-east Virginia, Postgres 18), reachable from Claude Code either through Neon's official MCP server or a plain connection string.

Claude was asked to act as technical lead — Senior Data Scientist / ML Engineer / Data Quality Specialist — delivering the work in inspectable phases, with English code and Portuguese-language analysis narration, questioning data quality before any modeling.

Critical finding — target granularity

Average_Score, the intended modeling target, turned out to be a hotel-level constant: only 34 distinct values across 1,492 hotels, fixed within each hotel, and correlated 0.96 with the per-hotel mean of the row-level Reviewer_Score. Modeling it row-by-row would mean pseudo-replication and near-guaranteed leakage from other hotel-level columns. This single finding reshaped the entire modeling plan — see Panel design.

Decision

Reframing the modeling target: a hotel × period panel

Agreed

Resolves the leakage trap above while answering the two questions the team actually wants answered: how did a given hotel's satisfaction move over time, and how could a similar hotel catch up.

Since Average_Score cannot move over time (it's a single 2017 snapshot per hotel), the target is reconstructed as the quarterly or semi-annual mean of Reviewer_Score — which does move, with a confirmed real within-hotel standard deviation of ≈0.20–0.30 points. NLP-derived aspect features (sentiment on cleanliness, staff, location, food, value, noise, facilities…) aggregate up from the review level to feed both a panel regression (what moves the score) and a peer-benchmarking engine (cluster similar hotels, diff a lagging hotel's aspect profile against top performers in its cluster).

Quarter vs. semester — which one supports a "did it improve?" claim? A formal Welch's t-test (first vs. last period, per hotel) was run on both granularities:

GranularityMedian reviews / cellHotels testableSignificant change (p<0.05)Smallest reliably detectable Δ
Quarter (n≥10)211,31912.8%1.04 pts
Semester (n≥10)361,38612.3%0.86 pts
Semester (n≥20)361,19714.0%0.72 pts

Conclusion: yes — "this hotel improved with statistical significance" is a defensible claim, and the semester grain at n≥20 is the better basis for that specific claim: it detects real improvements roughly 30% smaller than the quarterly cut, because larger per-cell samples shrink the standard error of the reconstructed mean. Quarter is kept as the finer trend-line grain for charts; semester is the unit for the formal significance test.

Decision

NLP strategy: free, local, review-tuned models

Validated

Paid API-based rewriting/extraction was reconsidered mid-session in favor of a free, fully local stack — zero cost and no review text ever leaves the machine.

PurposeModelValidated result
Overall review sentimentnlptown/bert-base-multilingual-uncased-sentiment1–5 stars; correctly separated positive vs. negative samples from the real dataset
Aspect-level sentiment (ABSA)yangheng/deberta-v3-base-absa-v1.1Correctly split cleanliness / staff / breakfast on a mixed test sentence (100%, 99%, 100% confidence)

The ABSA model invents a sentiment for aspects a review never mentions, so aspects are keyword-gated before scoring — an 8-aspect lexicon (cleanliness, staff, location, comfort, food, value, noise, facilities) in src/nlp_pipeline.py only unlocks the model for aspects actually referenced in the text. The full pipeline was run end-to-end on CPU samples (40 and 500 rows) to confirm correctness before handoff to a GPU machine for the full 515k-review run (estimated 1–3 hours).
